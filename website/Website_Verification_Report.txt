1. Introduction
The internet has become central to daily life — people use it for banking, shopping, education, healthcare, and communication. However, the same web that empowers also exposes users to serious threats like phishing sites, scam portals, and malware-distributing pages.

This project, Website Genuineness Verification System, aims to build an automated tool that analyzes any given URL and determines whether the website is trustworthy or potentially malicious. It does this by combining multiple detection signals — SSL validation, domain age analysis, blacklist checks, IP intelligence, and content scanning — into a unified trust score.

1.1 Motivation
• Over 1.5 million phishing websites are created every month globally.
• Many users, especially first-time internet users, cannot distinguish fake sites from real ones.
• Existing tools are either paid, API-limited, or too technical for general users.
• There is a need for a simple, open, fast verification tool usable by anyone.

1.2 Scope
This project covers the design, development, and deployment of a web-based verification tool. It will accept any URL as input and return a structured analysis report with a trust score and verdict.

2. Problem Statement & Objectives
The core problem is:

"Given a URL entered by a user, automatically determine whether the website is genuine, suspicious, or malicious — and explain why."

2.1 Objectives
1. Accept any URL as input via a web interface.
2. Perform multi-layered verification checks automatically.
3. Compute a weighted trust score based on all checks.
4. Display a clear verdict: Genuine, Suspicious, or Malicious.
5. Provide detailed findings for each verification layer.
6. Store results for historical lookup and reporting.

2.2 Expected Output
• Trust Score (0–100) with color-coded rating
• Verdict label: Genuine / Suspicious / Malicious
• Breakdown of each check (SSL, WHOIS, Blacklist, Content, IP)
• Downloadable verification report

3.2 Request Flow
7. User enters URL in the web interface.
8. Flask backend receives the URL and initiates verification pipeline.
9. Each verification module runs in parallel or sequence.
10. Results are aggregated into a Reputation Score.
11. Final verdict and report are sent back to the user.

3.3 Verification Pipeline
• Step 1: URL Parser — extracts domain, TLD, subdomain, protocol
• Step 2: SSL Checker — validates certificate issuer, expiry, authenticity
• Step 3: WHOIS Lookup — retrieves domain age, registrar, owner info
• Step 4: Blacklist Check — queries VirusTotal, PhishTank, Google Safe Browsing
• Step 5: Content Analyzer — scans page HTML for phishing keywords & scripts
• Step 6: IP Intelligence — checks hosting provider, VPN/proxy flags, geolocation
• Step 7: Score Engine — computes weighted trust score and verdict

4. ER Diagram / Database Design
The system uses a relational database. The following entities and relationships define the data model:

4.1 Entity Descriptions
EntityDescriptionUSERStores user details — name, email, role, and registration dateURLRepresents a website being verified — raw URL, domain, TLD, protocol, timestampsVERIFICATION_REQUESTTracks each verification request — links user to URL, stores verdict and trust scoreSSL_CERTIFICATEStores SSL cert details — issuer, validity dates, whether self-signed or validDOMAIN_INFOWHOIS data — registrar, registration date, expiry, domain age, countryBLACKLIST_CHECKResults from blacklist sources — threat type, whether URL is listedCONTENT_ANALYSISPage content scan — phishing patterns, malicious scripts, spam scoreIP_INFOIP address details — ISP, country, VPN/proxy/Tor node flagsREPUTATION_SCOREAggregated scores per module — final trust score and trust levelREPORTGenerated report for a verification request — summary, findings, recommendations
4.2 Key Relationships
• USER submits one or more VERIFICATION_REQUESTs
• VERIFICATION_REQUEST targets one URL
• URL has one SSL_CERTIFICATE, one DOMAIN_INFO, one CONTENT_ANALYSIS, one IP_INFO
• URL is checked against multiple BLACKLIST sources
• URL has one REPUTATION_SCORE (aggregated from all checks)
• VERIFICATION_REQUEST generates one REPORT

Note: The full ER Diagram in Mermaid format is provided as a separate file (website-verification-diagram.mermaid) alongside this report.
